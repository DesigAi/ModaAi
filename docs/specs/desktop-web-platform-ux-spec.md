# ModAI Team Desktop Web Platform: UX/UI specification

Дата: 2026-06-16

## 1. Назначение документа

Этот документ описывает desktop/web-версию ModAI Team: рабочий кабинет клиента fashion/apparel-направления для покупки пакетов, создания AI-продакшена, хранения образов и скачивания результатов.

Документ объединяет:

- текущую бизнес-модель ModAI: 7 фото и production-комплекты;
- клиентский pipeline из приложения;
- лучшие UX-идеи внешней desktop-спецификации: `Образ`, `Библиотека`, `Результаты`, понятные статусы;
- недостающие детали для реальной дизайн- и frontend-спеки: экраны, компоненты, состояния, CTA, empty/error/loading.

## 2. Продуктовая модель

ModAI Team desktop web продает пакеты кредитов, привязанные к понятным клиенту результатам:

| Продукт | Цена | Что получает клиент |
| --- | ---: | --- |
| `7 Photo Set` | 5 USDT | 1 результат: 7 фото, 0 видео |
| `3 Production Kits` | 100 USDT | 3 комплекта: 21 фото, 3 видео |
| `30 Production Kits` | 1500 USDT | 30 комплектов: 210 фото, 30 видео |

Один production-комплект = 7 фото в выбранном стиле поз + 1 видео по выбранному 15-секундному видео-шаблону.

Основное обещание интерфейса:

`Загрузите 2-7 фото одежды, выберите модель, локацию и позы - получите 7 модных фото, а в production-комплекте еще и 15-секундное видео.`

### 2.1 Кредиты и списания

В интерфейсе и backend-логике используются только два типа кредитов:

- `Фото-кредиты` - количество доступных запусков формата `7 фото`.
- `Комплект-кредиты` - количество доступных запусков production-комплекта `7 фото + 1 видео`.

Видео не является отдельным продаваемым кредитом. В интерфейсе нельзя показывать `осталось видео` как независимую величину, чтобы не создавать ожидание покупки или списания видео отдельно от комплекта.

Правила начисления:

| Покупка | Начисление |
| --- | --- |
| `7 Photo Set` | `photo_set_credits + 1` |
| `3 Production Kits` | `kit_credits + 3` |
| `30 Production Kits` | `kit_credits + 30` |

Кредиты также могут начисляться вручную в admin backend:

- только администратором с admin-доступом;
- заинтересовавшимся пользователям;
- в рамках маркетинговой активности;
- как компенсация после технической ошибки.

Маркетинговое ручное начисление пользователь видит как начисленный пакет `7 фото`: `photo_set_credits + 1`.
Если начисление связано с технической ошибкой или спорным результатом, служба поддержки сама решает, что вернуть или исправить: вернуть зарезервированный кредит, подтвердить исправленный результат, запустить ручное исправление или выдать компенсацию.

Действия службы поддержки в admin backend:

| Action | Что делает |
| --- | --- |
| `refund_credit` | возвращает зарезервированный кредит пользователю |
| `manual_fix` | фиксирует ручное исправление результата без нового списания |
| `grant_compensation` | выдает компенсационный кредит или пакет по решению поддержки |
| `confirm_spend` | подтверждает списание после ручной проверки или исправленного результата |

Доступные типы компенсации для `grant_compensation`:

| Compensation type | Что начисляет |
| --- | --- |
| `photo_set_credit` | 1 фото-кредит |
| `kit_credit` | 1 комплект-кредит |

Связь admin action и credit ledger:

| Admin action | Ledger event |
| --- | --- |
| `refund_credit` | `support_refund` |
| `manual_fix` | `support_manual_fix` |
| `grant_compensation` | `support_compensation` |
| `confirm_spend` | `spend_confirmed` |

После `confirm_spend` пользователь видит в результате service note `Результат подтвержден поддержкой`, а support badge скрывается.
После `refund_credit` пользователь видит service note `Кредит возвращен поддержкой`, support badge скрывается.
После `manual_fix` пользователь видит service note `Результат исправлен поддержкой`, support badge скрывается.
После `grant_compensation` пользователь видит service note `Компенсация начислена поддержкой`, support badge скрывается.

Правила списания:

- `7 фото`: `photo_set_credits - 1`.
- `production-комплект`: `kit_credits - 1`.
- После нажатия финального CTA на review-экране кредит резервируется на generation order.
- Пока кредит зарезервирован, он не считается доступным для нового production.
- Если order не был создан из-за ошибки валидации, оплаты или доступа, кредиты не списываются.
- Если generation order создан, но провайдер полностью не принял задачу, резерв откатывается.
- Списание подтверждается только после полного результата, прошедшего quality check: для фото-flow это 7 фото, для production kit-flow это 7 фото + 1 видео.
- Результат не выдается как частичный клиентский продукт: клиент получает только полный пакет.
- Если результат по нашей вине неполный, неуспешный или содержит технические артефакты, выполняется одна перегенерация за наш счет без нового списания. Если результат полный и прошел quality check, клиентская перегенерация не предлагается.
- Если вторая попытка также завершается ошибкой, неполным результатом или техническими артефактами, UI переводит результат в `support_required`.
- В состоянии `support_required` решение по зарезервированному кредиту принимает служба поддержки: подтвердить списание после ручного решения, вернуть кредит, запустить ручное исправление или выдать компенсацию.
- При переходе в `support_required` зарезервированный кредит получает ledger state `support_hold` до решения службы поддержки.
- При переходе в `support_required` система создает тикет в admin backend и отправляет уведомление в Telegram-бот сервисной службы.

Визуальное отображение кредитов:

- sidebar показывает доступные кредиты: `Фото-кредиты: N` и `Комплект-кредиты: N`; зарезервированные кредиты в доступный остаток не входят;
- раздел `Тарифы`: текущие остатки, историю платежей, историю резервов, списаний, возвратов и ручных начислений;
- review: явная строка `Будет зарезервирован: 1 фото-кредит` или `Будет зарезервирован: 1 комплект-кредит`.

Credit ledger states:

| State | Что значит |
| --- | --- |
| `grant` | кредит начислен после подтвержденной оплаты |
| `marketing_grant` | кредит начислен администратором вручную как маркетинговая активность |
| `support_compensation` | кредит или пакет начислен службой поддержки как компенсация |
| `reserve` | кредит зарезервирован после финального CTA на review |
| `reserve_release` | резерв снят, потому что order не создан или провайдер не принял задачу |
| `spend_confirmed` | кредит списан после полного результата и quality check |
| `support_hold` | кредит удержан до решения службы поддержки |
| `support_refund` | служба поддержки вернула кредит |
| `support_manual_fix` | служба поддержки вручную решила проблему без нового списания |

## 3. Ключевые клиентские сущности

### 3.1 Образ

`Образ` - клиентское название для подготовленной связки:

- утвержденная AI-модель;
- одежда/комплект клиента;
- аксессуары, если есть;
- preview модели в этом комплекте.

В MVP `Образ` является логической сохраненной связкой, а не отдельной платной генерацией. Он мапится на `BaseModel + ClothingKit + preview`.

Правило preview образа:

- до получения результата показывается summary: выбранная модель + комплект одежды;
- после получения результата preview образа становится первым кадром результата;
- отдельный dressed preview на шаге образа не используется.

Клиент не видит названия `Dressed Model Look Card`, `Outfit Card`, `prepared_token`, `Seedance prompt`, `job_id`.

Образ нужен для повторного использования:

- клиент создал 7 фото, образ сохранился;
- позже клиент может создать новый результат из этого образа без повторной загрузки одежды;
- production-комплект может использовать уже сохраненный образ.

Жизненный цикл образа:

| Статус | Что значит | Доступные действия |
| --- | --- | --- |
| `draft` | модель и одежда выбраны после начисления кредитов, но generation order еще не запускался | продолжить настройку, удалить |
| `ready` | образ сохранен и может запускать новые production | создать 7 фото, создать комплект, переименовать, удалить |
| `used` | по образу уже есть результаты | создать новый production, открыть связанные результаты |
| `archived` | образ скрыт из основной сетки, результаты не удалены | восстановить, удалить окончательно при наличии права |

Важно: создание `Образа` до финального review не должно списывать кредиты. Платная операция начинается только при запуске generation order.

### 3.2 Референсы одежды

На один новый production клиент использует минимум 2 и максимум 7 фото одежды. В текущем комплекте для production всего 7 слотов: 2 обязательных и 5 дополнительных.

Правило подачи в UI:

- фото 1 = front / перед изделия;
- фото 2 = back / зад изделия;
- фото 3-7 = пять дополнительных фото: детали одежды, вышивка, швы, ткань, фурнитура, посадка, аксессуары.

Гардероб является библиотекой одежды, аксессуаров и готовых комплектов. При запуске конкретного production из гардероба выбирается или собирается один комплект, и для этого production действует ограничение 2-7 фото. Внутри системы роли сохраняются как `front`, `back`, `detail`, `accessory`. Больше 7 фото в одном production payload не принимается.

### 3.3 Результат

`Результат` - готовая или текущая генерация:

- `7 фото`;
- `production-комплект`;
- видео-результат, если тариф включает видео.

7 фото всегда отображаются одной карточкой результата, а не россыпью отдельных файлов.
Клиентский результат всегда пакетный: либо полный `7 фото`, либо полный `7 фото + 1 видео`.

### 3.4 Пользователь и доступ

Desktop web требует авторизации до доступа к библиотеке, результатам и оплате.

MVP auth model:

- основной вход: magic link по email;
- порядок первого входа: сначала пользователь выбирает язык интерфейса RU или ES, затем вводит email для magic link;
- выбранный язык сохраняется в профиле пользователя и применяется ко всему сайту;
- после ввода email показывается состояние `Письмо отправлено` с masked email и кнопкой повторной отправки после cooldown;
- в состоянии `Письмо отправлено` доступно дополнительное действие `Изменить email`, которое возвращает пользователя к вводу email и сбрасывает текущий cooldown;
- если magic link истек, пользователь видит `Ссылка устарела` и CTA `Отправить новую ссылку`;
- если magic link уже использован или невалиден, пользователь видит `Не удалось войти по этой ссылке` и CTA `Получить новую ссылку`;
- после успешного входа used link нельзя использовать повторно;
- после входа frontend получает `user_id`, имя/username, доступные кредиты и историю;
- если пользователь открыл прямую ссылку без сессии, он видит auth screen, затем возвращается на исходный route;
- если пользователь потерял сессию после оплаты, он после входа возвращается к доступным кредитам и может продолжить production.

Auth screen не должен быть маркетинговой landing page. Это короткий рабочий экран: логотип, заголовок `Войдите в ModAI Team`, доступные методы входа, ссылка на документы.

## 4. Основная навигация desktop

Desktop web использует левый sidebar и рабочую область справа.

Разделы:

- `Создать`
- `Образы`
- `Результаты`
- `Тарифы`
- `Настройки`

Sidebar:

- верх: логотип/название `ModAI Team`;
- ниже: навигация;
- низ: остаток `Фото-кредиты` и `Комплект-кредиты`, профиль пользователя;
- primary shortcut: `Новый продакшен`.

Если кредитов нет, primary shortcut `Новый продакшен` ведет в `Тарифы`.

Для ширины меньше 900 px sidebar сворачивается в icon rail. Для tablet/mobile web используется bottom navigation, но основная версия документа проектируется под desktop.

## 5. Информационная архитектура

### 5.1 Создать

Главный рабочий раздел. Здесь пользователь запускает новый production.

Варианты старта после начисления кредитов:

- если есть `photo_set_credits`: доступен только фото-flow `7 фото`;
- если есть `kit_credits`: доступен только production kit-flow `7 фото + 1 видео`;
- если есть сохраненный образ и подходящие кредиты: `Использовать сохраненный образ`.

Если кредитов нет, раздел `Создать` показывает CTA `Выбрать тариф` и ведет в `Тарифы`. До оплаты и начисления кредитов загрузка референсов и создание production заблокированы.

### 5.2 Образы

Раздел для сохраненных образов и гардероба. Это библиотека повторного использования: здесь пользователь управляет готовыми образами, одеждой, аксессуарами и комплектами.

Функции:

- посмотреть образы плиткой;
- открыть образ крупно;
- переименовать образ;
- удалить образ;
- создать новый образ;
- начать новый production из образа;
- открыть вкладку `Гардероб`;
- посмотреть одежду, аксессуары и готовые комплекты;
- переименовать, скрыть или удалить элементы гардероба по правилам использования.

CTA `Создать образ` доступен только при наличии `photo_set_credits` или `kit_credits`. Если кредитов нет, CTA ведет в `Тарифы` через состояние `Недостаточно кредитов`.

Структура раздела:

- tab `Образы`;
- tab `Гардероб`.

Вкладка `Гардероб` не запускает генерацию сама по себе. Она нужна для управления библиотекой и подготовки комплектов, которые потом выбираются в flow `Создать`.

Карточка образа:

- preview;
- название;
- модель;
- комплект/название одежды;
- дата создания;
- quick actions: `Открыть`, `Создать 7 фото`, `Создать комплект`.

Правила quick actions:

- `Создать 7 фото` показывается активной только при наличии `photo_set_credits`;
- `Создать комплект` показывается активной только при наличии `kit_credits`;
- если нужных кредитов нет, action ведет в `Тарифы` через состояние `Недостаточно кредитов`.

### 5.3 Результаты

Раздел для готовых и текущих генераций.

Функции:

- список/сетка результатов;
- фильтры: `Все`, `7 фото`, `Комплекты`, `В процессе`, `Ошибка`;
- открыть результат;
- скачать ZIP;
- скачать видео только для production-комплекта, если `video_url` присутствует;
- создать новый production из того же образа.

Карточка результата:

- preview;
- тип результата;
- название;
- статус;
- дата;
- локация;
- пакет поз;
- наличие видео;
- primary action.

### 5.4 Тарифы

Раздел оплаты, доступа и истории.

Функции:

- показать credit summary;
- показать остаток: фото-кредиты и комплект-кредиты;
- купить пакет;
- показать PaymentIntent;
- показать историю платежей;
- показать историю списаний.

### 5.5 Настройки

Легкий раздел без админской сложности.

Функции MVP:

- профиль пользователя;
- язык интерфейса;
- уведомления;
- документы/правила;
- выход.

Настройки не дублируют тарифы и не становятся financial dashboard.

## 6. Desktop layout rules

### 6.1 Общий layout

Desktop viewport: `1440 x 900` как основной дизайн-таргет.

Структура:

- sidebar: 248 px;
- content max-width: 1180-1280 px;
- page header: title, subtitle, primary CTA;
- content grid: плотный, рабочий, без маркетингового hero;
- right inspector/drawer используется только там, где помогает review.

### 6.2 Визуальный стиль

Стиль: premium fashion production dashboard.

Требования:

- темная спокойная основа;
- золото/теплый акцент допустим, но не превращать весь UI в однотонную золотую тему;
- карточки радиусом 8 px или меньше;
- без декоративных градиентных орбов;
- без landing-page hero внутри рабочего продукта;
- реальные/сгенерированные fashion preview вместо абстрактных SVG-иллюстраций;
- плотность выше, чем в miniapp: desktop должен помогать сравнивать и управлять.

### 6.3 Компонентные правила

Использовать:

- tabs для фильтров;
- segmented controls для режима результата;
- cards для образов, результатов и тарифов;
- side drawer для просмотра образа/результата;
- modal только для оплаты, удаления, ошибок доступа и финальных success states;
- progress timeline для генерации;
- upload dropzone для одежды;
- icon buttons для quick actions;
- table/list для платежей.

Не использовать:

- длинные объясняющие тексты внутри каждого экрана;
- технические коды шаблонов в клиентском UI;
- вложенные карточки внутри карточек;
- disabled future-функции, если они не реализованы.

### 6.4 UI labels и internal names

Спецификация может использовать internal names для разработки, но пользовательский интерфейс должен показывать клиентские названия.

Языковое правило:

- клиентский интерфейс MVP показывается на русском;
- вторым клиентским языком закладывается испанский;
- английский используется только для internal/API names, технических ID и разработки;
- в пользовательском UI не показывать английские служебные слова вроде `secondary description`, `BaseModel`, `ClothingKit`, `GenerationOrder`, `scene_id`.

| UI label | Internal/API name |
| --- | --- |
| `7 фото` | `photo_set` |
| `Production-комплект` | `production_kit` |
| `Фото-кредиты` | `photo_set_credits` |
| `Комплект-кредиты` | `kit_credits` |
| `Модель` | `BaseModel` |
| `Комплект одежды` | `ClothingKit` |
| `Образ` | `Look` / saved model+kit bundle |
| `Результат` | `GenerationOrder` / result package |
| `Локация` | `scene_id` |
| `Пакет поз` | `pose_pack_id` |
| `Видео-шаблон` | `video_template_id` |

В UI не использовать: `BaseModel`, `ClothingKit`, `GenerationOrder`, `job_id`, `prepared_token`, `scene_id`, `pose_pack_id`, `video_template_id`.

## 7. Главный production flow

Desktop должен поддерживать два входа:

1. Первый production с нуля.
2. Быстрый production из сохраненного образа.

### 7.1 Первый production с нуля

Шаги:

1. Тариф: выбрать пакет.
2. Оплата: создать PaymentIntent и подтвердить оплату.
3. Начисление кредитов.
4. Формат определяется купленным пакетом и начисленными кредитами: `photo_set_credits` запускает фото-flow `7 фото`, `kit_credits` запускает production kit-flow `7 фото + 1 видео`.
5. Модель: выбрать сохраненную модель или создать новую.
6. Гардероб: загрузить 2-7 референсов, классифицировать ассеты и собрать комплект.
7. Образ: сохранить логическую связку модели и одежды как ready-образ без списания.
8. Локация: выбрать категорию, подвариант и controlled настройки локации.
9. Позы: выбрать один из готовых пакетов.
10. Видео: только для production-комплекта, выбрать один из 4 шаблонов.
11. Review: показать итоговый brief и списание кредитов.
12. Запуск генерации.
13. Progress.
14. Result.

До начисления кредитов шаги `Модель`, `Гардероб`, `Локация`, `Позы`, `Видео`, `Review` недоступны.

### 7.2 Быстрый production из образа

Шаги:

1. Выбрать сохраненный образ.
2. Если кредитов нет - пройти `Тариф` и `Оплату`.
3. Формат определяется доступным кредитом: `photo_set_credits` запускает `7 фото`, `kit_credits` запускает `7 фото + 1 видео`.
4. Выбрать локацию.
5. Выбрать позы.
6. Если доступен production-комплект - выбрать видео-шаблон.
7. Review.
8. Запуск.

Этот путь должен быть заметно короче первого. Он нужен для удержания и повторных продаж.

### 7.3 Доступ после оплаты

Правила как в miniapp:

- создание production требует оплаченных кредитов или free access;
- до оплаты загрузка референсов и создание generation order заблокированы;
- если кредитов нет, пользователь возвращается в `Тарифы`;
- после подтверждения платежа система начисляет кредиты;
- после начисления кредитов пользователь проходит шаги модели, гардероба, локации, поз и видео;
- `kit_credits` всегда включает 7 фото и 1 видео; отдельного выбора `только фото` внутри production-комплекта нет;
- `photo_set_credits` всегда включает только 7 фото; видео-шаг для него недоступен.
- В одной пользовательской сессии может быть только один активный production flow. Если flow уже запущен, кнопка `Новый продакшен` ведет в текущий незавершенный flow, а не создает второй параллельный.
- После нажатия финального CTA и резерва кредита новый flow нельзя начать, пока текущий order не перешел в `ready`, `failed` или `support_required`.

### 9.1 Шаг `Модель`

Цель: выбрать или настроить базовую AI-модель для съемки.

UI:

- preview модели;
- card `Создать новую модель`;

- действия: `Сохранить модель`, `Выбрать модель`;
- действие `Перегенерировать preview` доступно, если пользователь сам нашел технический артефакт в preview.

Параметры создания модели:

| Поле | Значения |
| --- | --- |
| Пол | Женский / Мужской / Унисекс |
| Возраст | 18-22 / 23-28 / 30-35 |
| Этнотип | Европейский / Азиатский / Латино / Афро / Смешанный |
| Комплекция | Стройная / Атлетичная / Песочные часы / Полная |
| Тон кожи | выбранный тон кожи |
| Заметки по модели | короткое текстовое уточнение без персональных данных реального человека |

Preview модели:

- генерируется на нейтральном фоне;
- модель показывается в черном обтягивающем боди и легинсах;
- если preview корректный, пользователь утверждает его и не перегенерирует по вкусу;

- кнопка активна максимум 3 итерации в рамках текущего создания модели;
- после третьей итерации дальнейшая генерация preview в этой сессии недоступна;
- чтобы снова пройти генерацию модели, пользователь начинает шаг создания модели заново;
- после утверждения модель сохраняется и может использоваться повторно.
- preview модели не списывает `photo_set_credits` или `kit_credits`; production-кредиты списываются только при запуске generation order на review.

Обязательные состояния:

- empty: `У вас пока нет сохраненных моделей. Создайте первую модель.`
- loading preview;
- ошибка генерации;
- approved;
- selected.

UI copy:

- Не использовать `BaseModel` в клиентском тексте.
- Использовать `Модель`.
- Не предлагать загрузку фото реального человека для клонирования внешности без отдельного consent-flow.

### 9.2 Шаг `Гардероб и комплект`

Цель: выбрать или собрать комплект в гардеробе, загрузить одежду/аксессуары и классифицировать ассеты.

UI:

- drag-and-drop зона;
- file picker;
- счетчик `0 / 7`;
- подсказка `Минимум 2 фото: перед и зад. До 7 фото можно использовать для деталей и аксессуаров.`;
- preview grid из 7 слотов;
- первые два слота подписаны `Перед` и `Зад`;
- остальные пять: `Деталь`, `Деталь`, `Деталь`, `Аксессуар`, `Аксессуар`;
- библиотека гардероба для одежды, аксессуаров и готовых комплектов;
- поле `Название комплекта` с обычным клиентским названием.

Для каждого загруженного ассета приложение определяет или запрашивает:

| Поле | Значения |
| --- | --- |
| Тип | одежда / аксессуар |
| Категория | dress / suit / top / bottom / outerwear / belt / glasses / hat / bag / shoes / jewelry |
| Ракурс | front / back / side / closeup / generic |
| Название | обычное клиентское название, например `Черное шелковое платье` |
| Принадлежность | новый или существующий `ProductItem` / `AccessoryItem` |

`ClothingKit` собирается из одного или нескольких `ProductItem`, нуля или нескольких `AccessoryItem` и клиентского имени комплекта.
Internal slug для backend генерируется автоматически и не показывается пользователю.

Гардероб как библиотека поддерживает:

- просмотр всех `ProductItem`, `AccessoryItem` и `ClothingKit`;
- создание нового `ProductItem` из загруженных ассетов;
- создание нового `AccessoryItem` из загруженных ассетов;
- редактирование названия, категории и принадлежности ассета;
- сборку `ClothingKit` из существующих `ProductItem` и `AccessoryItem`;
- переименование `ClothingKit`;
- удаление неиспользуемых ассетов и комплектов;
- повторное использование сохраненного `ClothingKit` в новых production.

Правила удаления:

- если asset или kit уже использован в `Образе` или `Результате`, удаление из библиотеки не удаляет уже созданные результаты;
- используемый asset можно скрыть из библиотеки, но его ссылка сохраняется внутри существующих результатов;
- скрытый asset не показывается в обычной сетке гардероба и не предлагается для новых комплектов;
- скрытые asset/kit доступны через фильтр `Скрытые`;
- скрытый asset/kit можно восстановить, если у пользователя есть право редактирования библиотеки;
- неиспользуемый asset/kit можно удалить окончательно после confirm modal;
- при удалении kit UI показывает confirm modal;
- если kit используется в сохраненном образе или результате, UI вместо удаления предлагает `Скрыть из библиотеки`.

Карточка гардероба:

- preview;
- тип: одежда / аксессуар / комплект;
- название;
- категория;
- наличие front/back;
- дата создания;
- actions: открыть, переименовать, добавить в комплект, скрыть/удалить.

Правило ракурсов:

- слот 1 автоматически получает ракурс `front`;
- слот 2 автоматически получает ракурс `back`;
- слоты 3-7 классифицируются как `detail`, `accessory`, `side`, `closeup` или `generic`;
- пользователь не меняет роли первых двух слотов вручную.

Accepted reference types:

- фото изделия спереди;
- фото изделия сзади;
- flat lay;
- изделие на манекене;
- изделие на человеке, если лицо не является главным объектом кадра;
- крупные детали: вышивка, швы, ткань, фурнитура, бирки, аксессуары.

File rules:

- форматы: JPG, PNG, WEBP;
- максимум: использовать общий backend limit `MODAI_MAX_UPLOAD_MB`, если он доступен, иначе UI показывает нейтральное `Файл слишком большой`;
- порядок можно менять drag-and-drop внутри slot grid;
- каждый файл можно удалить;
- загрузку с ошибкой можно повторить с того же слота;
- duplicate file не блокируется, но UI показывает warning `Похоже, это фото уже добавлено`.

Validation:

- меньше 2 фото: primary CTA disabled;
- больше 7 фото: лишние не принимаются, показывается inline warning;
- неподдерживаемый формат: показать inline error;
- слишком большой файл: показать inline error.
- если у одежды нет `front`-ассета: показать inline warning;
- если у одежды нет `back`-ассета: показать inline warning;
- если аксессуар плохо классифицирован или не имеет достаточного качества: показать inline warning.

CTA:

- `Продолжить`
- дополнительное действие: `Очистить`

Internal recovery:

- частичный результат не показывается как клиентский результат;
- если первая попытка по нашей вине неполная, неуспешная или содержит технические артефакты, система запускает одну перегенерацию за наш счет;

- клиент не получает кнопку скачивания неполного пакета.

Quality check считается пройденным только если:

- для фото-flow есть 7 файлов изображений;
- для production-комплекта есть 7 файлов изображений и 1 видео;
- ZIP-архив создан;
- ссылки на все файлы не битые;
- результат прошел модерацию качества.

Если quality check не пройден, кредит не подтверждается как списанный.

Если результат прошел внутреннюю проверку, но пользователь видит проблему в готовом результате, UI показывает CTA `Связаться с поддержкой`. Создается тикет в admin backend, Telegram-бот сервисной службы получает уведомление с пометкой `косяк на выходе`, дальше решение принимает служба поддержки. Карточка результата остается `ready`, файлы остаются доступными, дополнительно показывается badge `На проверке поддержки`.
Пометка `косяк на выходе` является только внутренним service tag для admin backend и Telegram-бота. Пользователь ее не видит.

| Статус | Что видно | Доступные действия |
| --- | --- | --- |
| `failed` | понятная пользовательская, валидационная или неустранимая причина без технических деталей | открыть, исправить данные или начать заново |

### 12.5 Result state/action matrix

`support_required` используется, когда order был принят, кредит был зарезервирован, но после автоматической перегенерации не получился полный качественный пакет.

| Статус | Что видно | Доступные действия |
| --- | --- | --- |
| `queued` | карточка с очередью и датой запуска | открыть, отменить если payload содержит `cancel_allowed: true` |
| `processing` | progress timeline, skeleton preview | открыть, вернуться к списку |
| `quality_check` | progress timeline, текст проверки | открыть |
| `archive_preparing` | готовые preview, архив готовится | открыть |
| `ready` | все 7 фото, видео для комплекта, ZIP | открыть, скачать ZIP, скачать видео, создать еще |
| `regenerating` | повторная генерация за наш счет после неполного результата, ошибки или технических артефактов | открыть |
| `support_required` | вторая попытка не дала полный качественный пакет | открыть, обратиться в поддержку |

Кнопка `Скачать видео` показывается только для production-комплекта со статусом `ready`, если `video_url` присутствует.

Support behavior:

- при переходе в `support_required` создается тикет в admin backend;
- в тикет передаются user, result/order, выбранный flow, модель, комплект, локация, позы, видео-шаблон если есть, причина quality check failure и история попыток;
- Telegram-бот сервисной службы получает уведомление о новом тикете;
- пользователь видит текст `Мы уже передали задачу в поддержку. Свяжемся с вами в течение дня.` и CTA `Связаться с поддержкой`;
- решение по кредиту принимает служба поддержки внутри admin backend.
